# Recoll 搜索引擎

## 项目介绍

本仓库包含用于构建 Docker 容器的脚本，该容器提供以下功能：
- Recoll 搜索引擎
- Recoll 搜索引擎的 Web 前端界面
- recollstatus 脚本，用于显示正在进行的 recollindex 会话状态
- Apache2 文件夹接口

多年来，我尝试过许多文档搜索引擎。大多数都没有 Web 前端界面，这大大限制了选择范围。最初我长期在家用实验室使用的是 Recoll。我喜欢它的原因是能够调整 Web 界面中结果的显示方式。

最近，由于我收集了超过 25,000 份文档（好吧，我是个数字囤积狂，收集了 12 年以上的文档），我放弃了 Recoll。放弃的原因是，通过 Xapian 引擎进行搜索变得越来越慢。后来我发现，这是因为我的桌面电脑和 Recoll 在争夺内存，导致了严重的换页（也就是电脑变慢）。

在最近一个月左右，我购买了几台迷你 PC。其中一台成为了我 350W 台式机的 40W 替代品。另一台成为了我的家庭实验室机器，运行在无头模式下。这台实验室机器的内存是旧台式机的两倍。我把大部分工具迁移到了实验室机器上，在此过程中重新尝试了 Recoll。没有了内存冲突，Recoll 再次变得可以接受了。

## 免责声明
* Recoll 的这个实例仅适用于家庭实验室使用。将其放到互联网上是个非常糟糕的主意，主要是因为容器的任何部分都没有内置安全功能。
* 如果您要索引超过 20,000 个文件，您可能会注意到查询文件时出现过度缓慢。如果发生这种情况，您可能需要考虑使用单独的机器作为专用主机。对于几千个文件，这将在笔记本电脑或台式机上运行得很好。
* 这是一个正在进行的工作。您可能会注意到"编辑"功能还不能正常工作。这是因为我正在重写修改源文档元数据的代码。在下面的截图中，第一个条目是包含额外元数据（例如标题、作者、标签）的 PDF 文件的搜索结果。第二个条目显示的是缺少元数据的 PDF 的搜索结果。简而言之，元数据是一个很好的功能，但不会影响搜索引擎或搜索结果。我也在考虑添加搜索标签的功能或服务。当然，我受限制的资源是"时间"。

## 安装说明

1) 编辑构建脚本以适应您的架构。
* 默认情况下，8084 是 Web 服务器的外部端口。它"屏蔽"了容器内的 80 端口（因为我已经在实验室机器上运行了另一个 80 端口的服务）。
* 8083 是外部和内部端口，您可以通过它访问 Recoll 的 WebUI。
* /data/docs 是您的文件在宿主机上的存储位置。
* /var/www/html/docs 是 Recoll 认为文件存储的位置（在容器内）。
* /data/recoll 是 Recoll 文件在宿主机上的存储位置。
* /root/.recoll 是 Recoll 认为其文件存储的位置（在容器内）。注意：这为搜索数据提供了持久性。换句话说，您不必每次停止/启动机器（或容器）时都重新索引文件。

2) 根据需要编辑 result.tpl 文件。result.tpl 文件是搜索结果中各个条目的模板。我已经根据个人使用习惯对其进行了调整。您可能想要撤销我的调整和/或添加您自己的修改。如果您想要原始版本，请访问下面的 docker-recoll-webui 链接。

3)
```
docker run -itd --name recoll -p 8084:80 -p 8083:8080 -v /data/docs:/var/www/html/docs -v /data/recoll:/root/.recoll ghcr.io/jiangchuanso/recoll-webui
```
4) 在浏览器中访问 http://localhost:8083。

您可以通过以下命令启动/停止容器（无需重新运行 build-image 或 build 脚本）：
```
docker stop recoll
docker start recoll
```

## 索引文件
您可以通过连接到容器并运行 "recollindex" 或 "recollindex -z" 命令来索引您的文件。后者将擦除现有索引并创建一个新索引。前者将扫描并向索引添加任何新文件（和/或从索引中删除缺失的文件）。如果您想检查正在进行的扫描的状态，请运行 "recollstatus" 命令。例如：
```
docker exec -it recoll bash
recollindex -z
recollstatus
```

第一个结果是包含嵌入元数据的文件（类似于 exiftool）。这些元数据包括：标题、作者和标签。第二个结果是缺少嵌入元数据的文件的结果。默认情况下，Recoll 包含从文件中提取元数据的功能，因此将它们添加到结果中很简单。

显示的日期表示我复制文件到文件夹的日期，而不是实际的出版日期。出版商的文件创建日期实际上在元数据中，如果需要，提取/显示该日期应该很容易。
